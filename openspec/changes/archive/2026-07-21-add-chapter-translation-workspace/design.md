## Context

当前 App 的编辑器位于 `ui/app/(app)/page.tsx`，顶部 `MenuBar` 由同一 `(app)` layout 持续展示。项目场景已经用 `IndexMap` 保存页面顺序，各页面文本节点也沿用检测后的阅读顺序；译文最终写入 `TextData.translation`，修补、渲染和导出均可继续消费现有数据。

现有 `pipeline::run` 是页级调度器，适合检测、OCR、修补和渲染，但 `llm` Engine 每次只接收一个页面。此次变更的核心约束是：当前项目视为一章，由用户选择对话式 API LLM、最大输出 token，并决定整章一次翻译或按固定页数串行分批；分批时通过此前全部经用户确认的摘要保持连续语义，不处理本地模型的上下文和显存限制。

## Goals / Non-Goals

**Goals:**

- 在当前 App 内提供清晰独立、与顶层菜单同级可达的章节翻译工作区。
- 在不重写通用流水线的前提下，复用现有页级检测、OCR、修补、渲染和任务事件。
- 将完整章节按确定顺序组织为单次请求或串行批次，并可靠映射回现有文本节点。
- 让后一批继承此前全部经用户确认的简短摘要，在不累积完整原文、译文或对话历史的情况下保持术语和语义连续性。
- 保留原编辑器作为翻译后的逐页精修和导出闭环。
- 让已有 `LlmGenerationOptions.max_tokens` 真正作用于支持的 API Provider。

**Non-Goals:**

- 不创建外部网页、第二个桌面窗口或独立在线服务。
- 不新增“卷/章”持久化实体；MVP 中当前项目的全部页面就是一个章节。
- 不支持选择部分页面、根据 token 自动决定批次大小、自动重试、多轮术语提取或流式输出；是否分批和每批页数完全由用户决定。
- 不支持本地 GGUF、DeepL、Google Cloud Translation 和 Caiyun。
- 不修改 `.khr` 格式，不替换现有单页翻译入口，也不新增导出格式。
- 不在章节翻译页面实现逐气泡编辑；所有精修继续在原编辑器完成。

## Decisions

### 1. 使用顶层直达入口和 App 内独立路由

在“处理”和“帮助”之间增加无下拉菜单的“章节翻译”按钮，已打开项目时导航到 `/chapter-translation`。新页面继续使用现有 `(app)` layout，因此窗口控制、顶栏、项目会话、事件连接和前端 Store 都保持不变；“返回编辑器”导航回 `/`。

页面采用三种状态：

- 配置态：显示项目名称和页数，提供 Provider、模型、目标语言、最大输出 token、作品背景/术语提示词、“分批翻译”开关和“每批页数”，以及“开始章节翻译”。最大输出 token 为必填正整数，首次默认 `32000`；分批默认关闭，启用后每批页数首次默认 `50`，两者随后沿用本次 App 会话中的最近设置。
- 运行态：按“准备页面”“章节翻译”“修补与渲染”展示阶段状态、`第 N/M 批` 进度和错误，不允许重复启动。
- 完成态：显示处理页数和结果摘要，主按钮为“返回编辑器精修”；导出统一在原编辑器完成。

备选方案是外部网页或全屏 Dialog。外部网页需要重复项目导入、状态同步和鉴权；Dialog 不利于长任务状态表达和浏览器返回行为。独立 App 路由改动较小且保留完整编辑闭环，因此采用该方案。

### 2. 增加专用章节长任务，不改变通用 Engine 作用域

新增 `POST /chapter-translations`，请求包含 `LlmTarget`、目标语言、`maxTokens`、可选 `brief` 和可选 `batchSize`，响应沿用 `{ operationId }`。`batchSize` 缺省或不小于总页数时只建立一个批次；其他正整数按页面顺序切分。任务类型记录为 `chapter-translation`，继续通过现有 jobs registry 和 SSE 发布状态。

后端任务按三个阶段编排：

1. 按项目页序根据 `Artifact::ready` 为每页补齐缺少的检测、分割、字体检测和 OCR Engine；已完整准备的页面不重新检测，完成一页立即上报页数进度。
2. 全部 OCR 就绪后创建章节快照，按 `batchSize` 建立一个或多个批次并串行调用 API LLM。每批成功后等待用户审核摘要，再把确认值追加到摘要历史。
3. 每批译文写回后，对该批需要的页面调用现有 inpainter 和 renderer，完成后再开始下一批。

这一方案只在新服务中组合多次已有 `pipeline::run`，避免把现有 `Engine` trait 改造成页级/项目级双作用域，也不会改变“处理全部页面”等旧入口的行为。

### 3. 使用结构化章节文档和任务内连续标识

章节快照按 `Scene.pages` 顺序遍历页面，再按 `text_nodes` 顺序遍历文本节点。每个单元保留 `PageId`、`NodeId`、原文和任务内连续字符串标识；发给 LLM 的用户消息使用 JSON 表示页面边界和文本块，例如页面对象包含 `pageIndex`、`pageName` 和 `blocks`，每个 block 包含 `id` 与 `source`。

系统提示由不可覆盖的中文固定规则与用户 `brief` 组合。批次请求的用户消息只包含当前批页面 JSON，以及按批次顺序排列的 `contextSummaries`；第一批不携带摘要。批量响应只允许返回 `translations: [{ id, text }]` 和当前批 `batchSummary`，非批量响应只允许 `translations`。解析器可接受纯 JSON 或单层 Markdown JSON code fence，但必须拒绝说明性文字、缺失项、重复项、未知项、空译文，以及批量模式缺失或空的 `batchSummary`。

第 N 批不会携带此前批次的完整原文、译文或聊天消息，而是携带第 1 至 N-1 批经用户确认的全部摘要。每批修补渲染完成后展示历史只读摘要和当前可编辑摘要，只有用户点击继续才固化当前值并开始下一批。备选方案是复用 Provider 的原生 conversation id 或持续追加 chat messages，但这会绑定特定 Provider并使历史内容持续膨胀，因此不采用。

采用任务内短 ID 而不是要求模型复述 UUID，可降低格式错误；后端查找表负责映射回 `PageId` 和 `NodeId`。备选方案是复用当前 `[N]` 文本格式，但它不能可靠表达页面边界，且现有解析器会把缺失项静默补为空字符串，因此不采用。

### 4. 每批完整校验后原子写回

每批 LLM 请求前保存该批目标节点的原文快照和全部已确认摘要检查点。响应解析完成后再次确认页面、节点和原文未被其他客户端修改，然后创建跨页面 `Op::Batch`，以一个历史记录写入当前批全部 `TextData.translation`。任何校验失败都不应用当前批，也不更新摘要历史；重试使用相同的摘要检查点。此前已完成的批次保持不变。

预处理和后处理仍沿用现有页级历史记录。当前批翻译写回成功但某页渲染失败时，任务停止在该批并显示“完成但有错误”，保留已写回译文和检查点，用户可以回编辑器修复或重跑该页。只有当前批翻译、写回和后处理均成功后，调度器才进入下一批。

### 5. 复用现有 Provider 生命周期并补齐 max_tokens

章节任务从现有 LLM catalog 过滤 `openai`、`openai-compatible`、`claude`、`gemini` 和 `deepseek`，后端使用同一允许列表再次验证。任务开始时使用所选 `LlmTarget` 加载 Provider，并把 `maxTokens` 合并到已有 `ProviderConfig.max_tokens`。

当前 `LlmLoadRequest.options` 和 `ProviderConfig.max_tokens` 已存在，但除 OpenAI-compatible 外尚未完整传递。实现时为 OpenAI、DeepSeek 的 Chat Completions 请求传入该值，为 Claude 替换硬编码值，为 Gemini 增加对应 generation config；未提供值的旧调用继续使用各 Provider 现有默认行为。这样无需新增第二套 Provider 抽象。

### 6. 前端仅保存工作区状态，不改变项目格式

新增轻量 `chapterTranslationStore` 保存表单快照、可选 `batchSize`、当前 operation id、总批次数、当前批次和页面状态，使用户在 App 内切换路由后可恢复运行进度。后端长任务在内存中持有已确认摘要检查点；API Key、Base URL 和模型目录继续来自现有 Settings 与 LLM catalog，章节提示词和摘要历史不写入 `.khr`。

章节完成页只返回原编辑器；导出继续调用原编辑器已有 rendered export，避免产生重复入口并绕过精修。

## Risks / Trade-offs

- [API 响应不完整或格式错误] → 严格解析并保持当前批译文原子写回；不推进摘要历史，使用同一检查点重试。
- [多批摘要发生漂移] → 固定用户提示词不可被模型修改，每批摘要必须先由用户审核；后续请求携带全部已确认摘要而不是只覆盖上一批。
- [Provider 对最大输出参数的支持方式不同] → 在各现有 Provider adapter 内转换，保持统一前端字段和后端验证。
- [长任务期间项目被 MCP 或其他客户端修改] → 写回前比较节点存在性与 OCR 原文快照，检测到变化则失败而不覆盖。
- [某页预处理失败导致整章无法翻译] → 在调用 API 前停止并列出失败页面，避免向 LLM 发送残缺章节。
- [章节页路由增加 UI 状态] → 仅增加一个内存 Store，不持久化新的项目数据；关闭 App 后沿用现有长任务生命周期限制。
- [章节翻译与现有全局 LLM 状态共享] → 依赖现有全局运行中任务互斥，章节任务启动后禁止并行加载或发起另一翻译任务。

## Migration Plan

1. 先上线 Provider `max_tokens` 透传和后端章节任务，保持旧 API 行为不变。
2. 生成 OpenAPI 客户端后加入章节翻译路由、Store 和顶层入口。
3. 通过多页 fixture 验证不分批单次请求、300 页按 50 页形成 6 个串行批次、全部已确认摘要继承、批次失败重试、严格回写、返回编辑器和导出回归。
4. 若需要回滚，可移除新入口、路由和任务服务；由于项目场景格式未变，已经写回的译文仍能由旧版编辑器正常读取。

## Open Questions

无。MVP 固定为“当前项目全部页面、默认单次 API LLM 请求、用户可设置固定每批页数、后一批继承此前全部已确认摘要、逐批写回并修补渲染、最终回编辑器精修”。
