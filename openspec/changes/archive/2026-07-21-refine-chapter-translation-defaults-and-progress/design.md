## Context

章节配置状态目前将 `maxTokens` 初始化为 `8192`。进度事件只有当前子 Pipeline 页索引和加权总体百分比，无法同时保存三个阶段各自的全章完成页数；翻译阶段又以一次 LLM 请求为原子操作。渲染器在没有显式 `TextStyle` 时会优先采用 `FontPrediction.text_color`，因此不同文本块可能出现不同颜色。

## Goals / Non-Goals

**Goals:**
- 默认使用 `32000` 最大输出 Token。
- 用真实图片数量表达准备、翻译、修补渲染进度，并在所有批次间累计。
- 章节翻译统一产生黑色初始译文，同时保留其他已有排版样式。

**Non-Goals:**
- 不把一次 LLM 请求拆成逐页请求，也不伪造 token 或逐页流式进度。
- 不移除原版字体颜色预测能力；该能力仍可用于普通 Pipeline。
- 不阻止用户在原编辑器对任意文本块重新调色。

## Decisions

1. 在 `PipelineProgress` 增加可选的 `chapterTotalPages`、`preparedPages`、`translatedPages` 和 `renderedPages`。通用 Pipeline 保持字段为空，章节任务的每个事件携带完整计数，保证 SSE 重连后仍能恢复三个阶段的显示。
2. 准备阶段按当前 Scene 中已满足章节前置 Artifact 的页面数计算，可复用的页面会直接计入；翻译阶段只在批次响应校验并写回后推进到该批结束页；后处理根据逐页 Pipeline tick 从上一批完成边界继续累计。
3. 运行页保留总体进度；准备与修补渲染卡片使用 `X / N` 和对应进度条，翻译卡片仅显示等待状态或批次状态，不再把加权总体百分比反算为阶段百分比。
4. `translation_op` 克隆已有 `TextStyle` 或创建默认样式，仅把 `color` 改为 `[0, 0, 0, 255]` 后随译文原子写回。相比在渲染器中加入章节专用分支，这能保证后续重新渲染和导出仍保持黑色，且精修修改可正常覆盖。

## Risks / Trade-offs

- [准备阶段复用缓存时可能很快显示满值] → 这是实际已准备图片数量，不增加虚假延时。
- [单批翻译仍会从批次起点跳到终点] → 等待期间明确显示活动状态和当前批次；只有拆分请求才能更细，但会损害上下文目标。
- [写入显式黑色会覆盖原有手动文字颜色] → 这是本次明确要求；其他样式保持不变，用户可在精修阶段重新调色。
