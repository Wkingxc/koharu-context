## Context

当前章节翻译复用逐页流水线，准备完成后只统计未满足 `Artifact::OcrText` 的页面数量；配置分别存在后端 `config.toml`、前端 Zustand 状态和系统钥匙串。需要在不改变原编辑器工作流和 `DataConfig.path` 语义的前提下，补齐可恢复错误、漫画级配置切换和无钥匙串密钥存储。

## Goals / Non-Goals

**Goals:**

- 让 OCR 缺失错误能直接定位到用户可编辑的页面与文本块。
- 用轻量本地预设保存并切换漫画处理相关设置。
- 将 API Key 写入当前数据目录的独立本地文件，避免 macOS 钥匙串弹窗。
- 保证重新编译或替换可执行文件不重置已保存的数据路径。

**Non-Goals:**

- 不自动跳过含空 OCR 的文本块，也不改变无文本节点页面的既有行为。
- 不在预设中保存 API Key、运行时目录、HTTP、外观或快捷键。
- 不实现预设的云同步、导入导出或跨设备迁移。
- 不迁移或删除操作系统钥匙串中的旧条目。

## Decisions

- 预设使用前端持久化 Store 保存：这些设置跨越后端 pipeline 配置和前端 LLM/翻译表单状态，应用时通过现有配置与 LLM API 同步后端。相较新增后端预设协议，该方案改动更小且能保持 API Key 完全独立。
- API Key 使用 `DataConfig.path` 下的专用 TOML 文件保存：这与用户选择的数据目录一致，并避免可执行文件替换影响密钥。`config.toml` 仍只写 `[REDACTED]`。
- 加载配置时把 `[REDACTED]` 归一为空，再由本地密钥文件注入真实值，避免缺少密钥文件时把脱敏占位符误认为有效 Key。
- OCR 诊断在准备阶段遍历页序和文本节点，生成稳定的人类可读错误；前端沿用任务错误通道并提供明确的“返回编辑器”操作。

## Risks / Trade-offs

- [前端本地预设不会自动跨机器同步] → 明确定位为本机快捷切换，并保持结构带版本迁移能力。
- [本地明文 Key 的保密性弱于系统钥匙串] → 使用独立文件、Git 忽略、Unix `0600` 权限，且接口与日志保持脱敏；这是用户明确选择的易用性权衡。
- [应用预设涉及多个写入步骤] → 后端配置和 LLM 加载成功后才标记当前预设，失败时显示错误。
- [旧钥匙串 Key 不再自动可用] → 首次升级后需重新输入一次，避免迁移读取再次触发钥匙串授权。

## Migration Plan

升级后不读取或删除旧钥匙串条目；用户首次保存 API Key 时创建本地文件。既有 `config.toml` 和 `DataConfig.path` 原样加载。回滚代码后旧版本仍可使用原钥匙串条目，但不会读取新本地文件。
