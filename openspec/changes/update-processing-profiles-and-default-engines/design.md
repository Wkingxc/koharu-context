## Context

处理配置存储在 UI 的持久化 Zustand store 中，保存动作会创建静态快照；引擎默认值由 Rust `PipelineConfig::default()` 提供，并通过 `serde(default)` 兼容旧配置。此次变更同时涉及 UI 预设持久化和后端默认配置，但不需要新增 API。

## Goals / Non-Goals

**Goals:**
- 让用户明确触发当前配置的覆盖更新。
- 保留配置的 `id`、`name` 和创建时间，仅替换设置快照。
- 为全新安装提供截图中指定的默认引擎组合。

**Non-Goals:**
- 不做设置变化后的自动同步。
- 不新增配置重命名、历史版本或自动迁移已有引擎选择。

## Decisions

- 在 store 增加按 `id` 替换配置的原子动作，更新时重新捕获当前设置，再保留原配置身份字段。相比删除后新增，这能保持菜单选中状态和配置顺序。
- 更新入口放在现有顶部“配置”菜单，并在无 `activeProfileId` 时禁用。相比自动保存，该方式避免用户临时试验设置时污染已保存配置。
- 仅修改 `PipelineConfig::default()`。已有完整 `config.toml` 会反序列化原值，因此不会被新默认值覆盖；字段缺失时按新默认补齐。

## Risks / Trade-offs

- [用户修改设置后仍显示当前配置已选中] → 更新动作由用户显式触发，文案说明其覆盖语义；本次不扩展脏状态检测。
- [已有不完整配置的缺失字段采用新默认值] → 这是 `serde(default)` 的预期兼容行为，完整配置不受影响。
