# Chapter Translation

### Requirement: 顶部同级入口与 App 内页面
系统 SHALL 在顶部菜单栏提供与“处理”“帮助”同级的“章节翻译”直达入口。该入口 MUST 在当前 App 窗口内打开独立章节翻译页面，并在未打开项目时禁用，不得启动外部浏览器或独立 Web 应用。

#### Scenario: 从编辑器进入章节翻译页面
- **WHEN** 用户已打开项目并点击顶部“章节翻译”入口
- **THEN** 系统在保留当前项目会话的情况下显示章节翻译页面

#### Scenario: 未打开项目
- **WHEN** 当前没有已打开项目
- **THEN** “章节翻译”入口处于禁用状态且不会发起任何任务

### Requirement: 翻译任务配置
章节翻译页面 SHALL 展示当前项目页数，并允许用户选择已配置的对话式 API LLM Provider、模型、目标语言、正整数最大输出 token、本次任务的作品背景和术语提示词，以及是否启用分批翻译。启用分批时，用户 MUST 能够设置正整数每批页数；系统 MUST 在用户明确点击“开始章节翻译”前不调用翻译 API。

#### Scenario: 配置并开始任务
- **WHEN** 用户完成必填配置并点击“开始章节翻译”
- **THEN** 系统使用页面中确认的 Provider、模型、目标语言、最大输出 token、提示词和可选每批页数创建任务配置快照并启动任务

#### Scenario: 仅进入配置页面
- **WHEN** 用户进入章节翻译页面但尚未点击开始按钮
- **THEN** 系统不发起检测、OCR、翻译、修补或渲染任务

### Requirement: 仅支持对话式 API LLM
章节翻译 SHALL 仅接受 OpenAI、OpenAI-compatible、Claude、Gemini、DeepSeek 等能够遵循上下文提示和结构化输出约束的 API LLM Provider。系统 MUST 拒绝本地模型以及 DeepL、Google Cloud Translation、Caiyun 等传统机器翻译 Provider。

#### Scenario: 选择可用 API LLM
- **WHEN** 用户选择已正确配置的对话式 API LLM 和模型
- **THEN** 系统允许用户启动章节翻译

#### Scenario: 非上下文翻译 Provider
- **WHEN** 请求携带本地模型或传统机器翻译 Provider
- **THEN** 后端拒绝请求并返回可理解的配置错误

### Requirement: 自动准备完整有序的章节文本
任务启动后，系统 SHALL 按当前项目页序处理全部页面；对于尚未具备完整 OCR 文本的页面，系统 MUST 复用现有检测和 OCR 能力补齐文本。只有全部页面准备成功后，系统才能按项目页序以及各页文本节点阅读顺序构造章节文本，其中每个文本块均具有可映射回 `PageId` 和 `NodeId` 的稳定任务内标识。

#### Scenario: 部分页面尚未识别
- **WHEN** 项目中存在缺少文本节点或 OCR 文本的页面
- **THEN** 系统先处理这些未就绪页面，并在全部页面就绪后构造章节翻译输入

#### Scenario: 页面准备失败
- **WHEN** 任一页面的检测或 OCR 无法完成
- **THEN** 系统停止在翻译 API 调用之前，保留已有项目数据并显示失败页面和原因

### Requirement: 单次或摘要继承分批翻译
系统 SHALL 支持不分批和用户指定页数分批两种模式。未启用分批，或总页数不大于每批页数时，系统 MUST 将完整章节文本作为一次 API LLM 请求；启用分批且总页数大于每批页数时，系统 MUST 按项目页序连续切分并顺序翻译。固定规则 MUST 始终包含目标语言、保持文本块标识、摘要历史输入和仅返回结构化结果等约束；用户提示词只能补充作品背景和术语，不得替换固定规则。最大输出 token MUST 直接应用于每一次 Provider 请求。

#### Scenario: 发起整章翻译
- **WHEN** 所有页面 OCR 文本准备完成且用户未启用分批，或总页数不大于每批页数
- **THEN** 系统按照页面边界和阅读顺序向所选模型发送且仅发送一次整章翻译请求

#### Scenario: 发起分批翻译
- **WHEN** 用户启用分批、设置每批 50 页且当前项目有 300 页
- **THEN** 系统按项目页序建立 6 个批次并依次翻译，不并行发起批次请求

#### Scenario: 继承全部已确认摘要
- **WHEN** 系统准备翻译第 N 批且 N 大于 1
- **THEN** 本次请求携带固定规则、用户提示词、第 1 至 N-1 批按顺序排列的全部已确认摘要和第 N 批原文，不携带此前批次的完整原文、译文或对话历史

#### Scenario: 审核并固化当前批摘要
- **WHEN** 第 N 批 API 响应通过结构校验
- **THEN** 响应同时提供第 N 批译文和 `batchSummary`；系统在该批译文成功写回和修补渲染后暂停，只有用户确认或修改摘要并点击继续，才把该摘要加入历史并开始下一批

#### Scenario: 使用任务提示词
- **WHEN** 用户填写人物关系、作品背景或术语说明
- **THEN** 系统在保留固定输出和映射规则的同时把这些说明加入整章翻译上下文

### Requirement: 逐批完整校验并原子写回
系统 MUST 校验每次 API 响应能够唯一、完整地覆盖当前批次的全部文本块，且不得包含重复或未知标识；批量模式还 MUST 包含非空 `batchSummary`。只有当前批次校验全部通过后，系统才能将该批译文以一个历史批次写回对应的 `TextData.translation`；校验失败时不得写入当前批次的部分译文，也不得推进摘要检查点。后续批次失败 MUST 保留此前已成功写回的批次。

#### Scenario: 当前批返回完整结果
- **WHEN** API 响应包含当前批全部且唯一的任务内文本块标识、非空译文，并在批量模式包含非空 `batchSummary`
- **THEN** 系统将当前批译文映射回对应页面和文本节点，生成一个可撤销的历史批次，并在用户确认后保存新的摘要检查点

#### Scenario: 当前批返回结果缺失或截断
- **WHEN** API 响应缺少文本块、包含重复标识、未知标识、空译文、批量模式缺少 `batchSummary` 或无法解析
- **THEN** 系统把当前批标记为失败，展示错误摘要，保持当前批原有译文不变，并允许使用同一组已确认历史摘要重新翻译当前批

#### Scenario: 后续批次失败
- **WHEN** 第 N 批失败且此前批次已经成功写回
- **THEN** 系统保留第 1 至 N-1 批的译文、修补渲染结果和已确认摘要检查点

### Requirement: 复用现有修补渲染并返回精修
每个批次译文写回成功后，系统 SHALL 复用现有逐页修补和渲染能力生成该批页面的项目内结果，再继续下一批。全部批次完成后，完成页面 MUST 仅提供“返回编辑器精修”主操作；返回编辑器后，用户仍可修改任意页面的 OCR、译文、文本框和渲染效果，并使用原编辑器导出。

#### Scenario: 翻译完成后精修
- **WHEN** 章节翻译、修补和渲染成功完成且用户点击“返回编辑器精修”
- **THEN** 系统返回原编辑器并显示同一项目中已写回的逐页结果

#### Scenario: 统一从原编辑器导出
- **WHEN** 章节翻译任务完成
- **THEN** 完成页不提供独立导出操作，用户返回原编辑器精修并导出渲染结果

### Requirement: 任务状态反馈
章节翻译页面 SHALL 展示准备页面、章节翻译、逐页修补渲染三个阶段的运行状态，以及当前批次、总批次数、成功或失败摘要。运行中的任务 MUST 防止重复启动；离开并返回该页面时，系统 SHALL 从现有任务状态恢复显示。

#### Scenario: 任务运行中
- **WHEN** 章节翻译任务仍在运行
- **THEN** 页面显示当前阶段和“第 N/M 批”进度，并禁用新的“开始章节翻译”操作

#### Scenario: 任务失败
- **WHEN** 任一阶段失败
- **THEN** 页面保留用户任务配置，显示失败阶段和原因，并允许用户修正配置后重新开始
